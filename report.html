Which LLM should score your CVEs?

A measured answer for vens — the tool that turns a scanner report plus your project context into OWASP risk scores. Two things decide it: does the model understand the CVE, and does it use your context? We tested both — and kept ourselves honest about where the test itself is weak.

The short answer

No single model wins everything, and the top of the table is a statistical tie. Pick by what you optimize for — cost, auditability, or coarse bulk triage.

Half one — does it understand the CVE?

Reusing the public CTIBench / CTI-VSP task: read a real CVE description, predict its CVSS v3.1 vector. Metric is MAD — mean absolute error against the NVD score over 200 CVEs, with the vector parsed by the cvss library so the model never does the arithmetic. Lower is better.

Read it as tiers, not a strict ranking. The top-4 (0.75–0.88) sit inside one standard error — calling gpt-5.5 “first” over sonnet-4-6 (0.006 apart) is noise; so is flash-lite over flash. Local models (free, via Ollama) aren’t there yet. deepseek-r1:8b 1.48, qwen2.5:7b 1.54, gemma2:9b 1.64, llama3.2 1.69 — all four land at or above the 1.57 “always 7.5” floor, and their confidence intervals overlap it: on this task they’re statistically indistinguishable from a constant guess, and they over-rate every CVE. On context they’re worse — only the reasoning model (deepseek) uses it at all, and jitterily; the others pin to “critical” regardless.

04

Where the LLM actually beats the rule engine

vens’s prompt already encodes most scoring as arithmetic on your context, so we built that rule engine explicitly — a non-LLM null model — and asked where the LLM genuinely beats it. It has to be a case the rule structurally cannot reach: reading the CVE text.

Clean wins — reading the CVE. Reachability stated only in the operator’s notes: the rule is blind to free text; 6/8 models move the score sharply (one drops an unreachable RCE from 81 to 9), while flash-lite and nano ignore the notes entirely. Accountability under low data sensitivity: the rule scores technical=2, but 8/8 raise it to 8–9 — and it survives de-telegraphing (see below), so it’s genuine reading, not a keyword.
The one that isn’t. The pure-DoS case looks like a win — models do score technical below the naive max — but the effect is small (~8 vs 9) and collapses when we strip the phrase “denial of service”: it was mostly keyword-matching, not real understanding of availability-only impact. On the arithmetic axes (exposure, sensitivity, criticality, compliance) the LLM only matches the rule, as it should; a per-model Wilcoxon is significant for only 3 of 8.
05

The honest part — where it (and the test) breaks

A benchmark that only shows wins is marketing. Four things every vens user — and reviewer — should hear.

Cheap models re-inflate, they don’t de-escalate

gemini-2.5-flash-lite keeps an unreachable, low-value Log4Shell at Critical (48.8) — worse than the dumb baseline’s 10/Low. It is the least context-sensitive model here: flat on the sensitivity ladder, flat on the reachability notes. Below a capability line, the model scores the CVE’s reputation, not your system.

A wrong context YAML yields a wrong score — by design

We labeled a real payment-auth-bypass as “internal / low-value.” The capable models trust you: sonnet 12, pro 16.5 — close to the rule baseline. That is spec-correct for vens: the context is the operator’s declared truth, garbage-in-garbage-out. It’s not silent, either — gpt-5.4-mini flags the contradiction in its written rationale while still following the numbers. The lesson is operational, not a model defect: your YAML is load-bearing.

Accurate isn’t stable

gpt-5.4-mini is the value champion but one of the least consistent — risk SD 3.2 across three identical, temperature-0 runs; gpt-5.4-nano is worse (3.7). For a VEX you’ll attest and ship, reproducibility is a feature, and there claude-sonnet-4-6 (SD 0.85) and the deterministic Gemini models win. Note: SD over n=3 is a coarse estimate, not a precise ranking.

We de-rigged the context test — and one discriminator failed it

The obvious objection: the discriminator answers are partly stated in the CVE text (“availability only”), so a model could keyword-match instead of reason. We built non-telegraphed twins — same context, but the CVE no longer names the impact class — and re-ran. Accountability holds (technical stays 8–9 without the word “audit”): real reading. DoS collapses (technical drifts back toward the naive max): mostly keyword-matching. So Half two now separates genuine context-reading from pattern-matching instead of assuming it. Remaining caveat: the 0–81 scale saturates, so “should rise” is measured in bands, not points.

Methodology, threats to validity & attribution

Design. Two halves. Severity understanding reuses the public CTIBench / CTI-VSP task (MAD vs NVD CVSS, 200 CVEs). Context modulation — vens’s own contribution, unlabeled anywhere public — uses contrastive scenarios through the vens CLI: real CVEs for the two showcases (credibility), synthetic CVEs for the directional families (to isolate the context variable from memorized severity). No gold label exists for context-conditioned risk, so we report direction, consistency, and discrimination — never absolute accuracy.

Statistics, honestly. Discriminator direction is reported by model over 8 models: accountability 8/8, DoS 7/8, reachability 6/8 (binomial p = .004 / .035 / .14). We report per-model counts, not a pooled sign-test p, because the CVE×model cells are correlated. The overall LLM-vs-baseline Wilcoxon is significant for 3 of 8 models (flash, pro, mini); flash-lite and nano diverge from the rule engine by a median of zero — no measurable value added. Threshold-sensitivity: the accountability conclusion survives all three threshold sets, the DoS one does not. Consistency SDs are n=3 sample standard deviations — qualitative.

Reproducibility limits. Models are named by floating July-2026 aliases; resolved snapshots were not recorded, so the ranking is pinned to weights that may move. The CTI-VSP prompt sends the CVE description twice (constant across all models — internal ranking unaffected, but not directly comparable to CTIBench’s published GPT-4=1.31 / Gemini-1.5=1.09). Part-B ran on an unreleased +dirty vens build. Confidence intervals are bootstrap over the 200 CVEs, computed where per-CVE predictions were saved (sonnet, mini, the three Gemini models, and the four local models); the top-4 “tie” rests on their overlapping intervals.

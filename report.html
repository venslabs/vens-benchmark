Which LLM should score your CVEs?

A measured answer for vens — the tool that turns a scanner report plus your project context into OWASP risk scores. Two things decide it: does the model understand the CVE, and does it use your context? We tested both — and kept ourselves honest about where the test itself is weak.

04

Where the LLM actually beats the rule engine

vens’s prompt already encodes most scoring as arithmetic on your context, so we built that rule engine explicitly — a non-LLM null model — and asked where the LLM genuinely beats it. It has to be a case the rule structurally cannot reach: reading the CVE text.

Clean wins — reading the CVE. A pure-DoS CVE with critical data: the naive rule maxes impact via data-sensitivity; 4/4 models read “availability only” and score it lower. Reachability stated in the operator’s notes: the rule is blind to free text; 3/4 models move the score, flash-lite doesn’t. These are the discriminators a rule can’t fake.
Not a clean win. The “accountability” case looks like a win, but a smarter rule (max over all declared requirement fields) also raises it — so it tests instruction-following, not reading. And on the pure arithmetic axes (exposure, sensitivity, criticality, compliance) the LLM only matches the rule — as it should; a per-model Wilcoxon is non-significant for 3 of 4 there.
05

The honest part — where it (and the test) breaks

A benchmark that only shows wins is marketing. Four things every vens user — and reviewer — should hear.

Cheap models re-inflate, they don’t de-escalate

gemini-2.5-flash-lite keeps an unreachable, low-value Log4Shell at Critical (48.8) — worse than the dumb baseline’s 10/Low. It is the least context-sensitive model here: flat on the sensitivity ladder, flat on the reachability notes. Below a capability line, the model scores the CVE’s reputation, not your system.

A wrong context YAML yields a wrong score — by design

We labeled a real payment-auth-bypass as “internal / low-value.” The capable models trust you: sonnet 12, pro 16.5 — close to the rule baseline. That is spec-correct for vens: the context is the operator’s declared truth, garbage-in-garbage-out. It’s not silent, either — gpt-5.4-mini flags the contradiction in its written rationale while still following the numbers. The lesson is operational, not a model defect: your YAML is load-bearing.

Accurate isn’t stable

gpt-5.4-mini is the value champion but the least consistent — risk SD 3.4 across three identical, temperature-0 runs (one cell swung 17→52). For a VEX you’ll attest and ship, reproducibility is a feature, and there claude-sonnet-4-6 (SD 0.75) wins. Note: SD over n=3 is a coarse estimate, not a precise ranking.

The context test is partly rigged in the LLM’s favor

Construct-validity caveat we won’t hide: the discriminator answers are partly stated in the CVE text (“availability only”, “not reachable”) and the null baseline is built to ignore that text — so Half two partly measures “reads text the rule skips,” which is close to tautological. It’s a real capability, but modest. Non-telegraphed variants, more models, and n≥10 repeats are the next run. Also: the 0–81 scale saturates (a third of high-context cells pin near the ceiling), so “should rise” is only measurable in bands; and one Gemini family is n=1, not n=3, from a mid-run credit outage.

Methodology, threats to validity & attribution

Design. Two halves. Severity understanding reuses the public CTIBench / CTI-VSP task (MAD vs NVD CVSS, 200 CVEs). Context modulation — vens’s own contribution, unlabeled anywhere public — uses contrastive scenarios through the vens CLI: real CVEs for the two showcases (credibility), synthetic CVEs for the directional families (to isolate the context variable from memorized severity). No gold label exists for context-conditioned risk, so we report direction, consistency, and discrimination — never absolute accuracy.

Statistics, honestly. Discriminator direction is reported by model (DoS 4/4, reachability 3/4, accountability 4/4 models); the pooled sign-test p is not used, because the CVE×model cells are correlated — real power comes from the pending 8-model run. The overall LLM-vs-baseline Wilcoxon is significant for 1 of 4 models (direction consistent). Threshold-sensitivity for reachability is invariant by construction (the rule never reads notes); the DoS discriminator does not survive one of the three threshold sets. Consistency SDs are n=3 sample standard deviations — qualitative.

Reproducibility limits. Models are named by floating July-2026 aliases; resolved snapshots were not recorded, so the ranking is pinned to weights that may move. The CTI-VSP prompt sends the CVE description twice (constant across all models — internal ranking unaffected, but not directly comparable to CTIBench’s published GPT-4=1.31 / Gemini-1.5=1.09). Part-B ran on an unreleased +dirty vens build. Confidence intervals are computed only where per-CVE predictions were saved (the Gemini models); the top-4 “tie” rests on the ~0.08 standard error, not full CIs.
